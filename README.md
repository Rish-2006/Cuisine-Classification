## Project Overview

This project builds and evaluates two multiclass classifiers to predict the **primary cuisine** of a restaurant from structured metadata features (location, price, ratings, delivery options). The pipeline covers:

- Data cleaning, outlier handling, and feature encoding
- Stratified 80/20 train/test splitting
- Two model implementations: **Logistic Regression** (baseline) and **Random Forest** (ensemble)
- Complete evaluation: Accuracy, Precision, Recall, F1-Score, Confusion Matrix
- Per-cuisine bias analysis identifying data imbalance issues

## Dataset

**Source:** Zomato Restaurant Dataset (`data/dataset.csv`)

| Property | Value |
|---|---|
| Raw rows | 9,551 |
| Raw features | 21 |
| Processed rows (after filtering rare classes) | 9,097 |
| Engineered features | 57 |
| Target classes (cuisines) | 33 |
| Target column | `primary_cuisine` (first cuisine listed in `Cuisines`) |

**Class distribution is highly skewed:**
- North Indian: ~33% of all rows
- Majority of remaining 32 classes: each < 5% of data

## How to Run

### Full pipeline (recommended)

```bash
python main.py
```

This runs all three steps: preprocessing → training (with 5-fold CV) → evaluation.

### Skip cross-validation (faster, ~20 seconds)

```bash
python main.py --no-cv
```

### Custom dataset path

```bash
python main.py --data /path/to/your/dataset.csv
```

### Run steps individually

```bash
# Step 1: Preprocess only
python src/data_preprocessing.py

# Step 2: Train only (requires processed.csv)
python src/train.py

# Step 3: Evaluate only (requires trained models)
python src/evaluate.py
```

## Key Findings

### Performance

| Metric | Logistic Regression | Random Forest |
|---|---|---|
| Accuracy | 0.3533 | 0.3330 |
| Macro Precision | 0.1212 | **0.2153** |
| Macro Recall | 0.1159 | **0.1660** |
| Macro F1 | 0.0978 | **0.1761** |
| Weighted F1 | 0.2255 | **0.2891** |

**Reference baselines:**
- Always predict "North Indian" (majority class): **0.3291**
- Random guess (1/33 classes): **0.0303**

### Why accuracy is bounded near the majority baseline

Both models sit close to the majority-class accuracy (~33%). This is **not a modelling failure** — it is a direct consequence of the available features:

1. **The primary signal is geographic:** Longitude and Latitude account for ~45% of Random Forest's feature importance. This makes sense — cuisine type correlates with geography — but the relationship is noisy and indirect.

2. **North Indian dominates at ~33% of records.** Any model trained on this data will be heavily biased toward predicting North Indian, since that is the most common label regardless of feature values.

3. **Most other cuisines have too few samples** to learn reliable decision boundaries. 15 out of 33 classes have fewer than 20 test samples.

4. **No text features are used.** The restaurant *name* is the strongest real-world signal for cuisine type ("Burger King" → Burger, "Sushi Palace" → Japanese), but it was deliberately excluded from structural features.

### Bias and imbalance observations

- **15 cuisines** have fewer than 20 test samples → models struggle to learn these
- **North Indian** alone is ~33% of the data → models default to it under uncertainty
- The RF's `balanced_subsample` class weight partially compensates, resulting in better Macro F1 than LR despite slightly lower raw accuracy
- LR achieves higher accuracy by predicting North Indian more often; RF trades accuracy for better coverage of minority classes

### Feature importances (Random Forest)

Top predictors in order:
1. **Longitude** — strongest geographic discriminator
2. **Latitude** — complements longitude for regional clusters
3. **Votes** — popularity proxy; correlates with cuisine category
4. **Average Cost for two** — price tiers correlate with cuisine style
5. **Aggregate rating** — quality signal

## Design Decisions

### Target derivation
The raw `Cuisines` column contains comma-separated multi-label values (e.g., "North Indian, Chinese, Mughlai"). We extract the **first listed cuisine** as the primary label. This is a reasonable proxy for the restaurant's main speciality while keeping the task in the standard multiclass framework.

### Rare class removal
Classes with fewer than 20 total samples are removed before splitting. With fewer than 16 training samples, a class cannot be reliably learned and will corrupt stratified splits.

### No `class_weight='balanced'` on Logistic Regression
Using balanced weights actually *reduces* accuracy for LR when the majority class is a strong signal. We use `class_weight='balanced_subsample'` on RF (per bootstrap sample) which is more stable for trees, and leave LR with uniform weights. Both choices are documented in code comments.

### StandardScaler before LR
Logistic Regression with `lbfgs` converges significantly faster and to a better solution when features are normalised. The scaler is embedded in a `Pipeline` so it is never accidentally applied to test data in isolation.

### OHE with top-N bucketing
City (1,500+ unique values) and Currency are bucketed to top-20 / top-10 most frequent values respectively, with all others mapped to `"Other"`. This prevents a dimensionality explosion while retaining the most informative categories.

## Future Improvements

1. **Add restaurant-name NLP features** — TF-IDF or embedding of restaurant names would be the single highest-impact improvement, as the name is a direct signal for cuisine type.

2. **Hierarchical grouping** — Reduce 33 classes to ~10 macro-categories (e.g., "Indian Subcontinent", "East Asian", "Western") to improve signal-to-noise ratio.

3. **XGBoost / LightGBM** — Gradient boosting typically outperforms RF on tabular data; a quick hyperparameter search with Optuna could yield gains.

4. **Oversampling** — `SMOTE` or `RandomOverSampler` targeted at minority classes (excluding North Indian) could improve Macro F1.

5. **Threshold calibration** — Adjust per-class decision thresholds to trade accuracy for better minority-class recall.
